# Study 2 — SDK drift longitudinal

> *Question*: How often does PX roll its SDK, and what changes between rolls?

## Status

Ongoing — automatically updated by `.github/workflows/sdk_drift_detection.yml`.

## Method

Each day, the workflow:

1. Re-downloads the iFood and Grubhub SDKs.
2. Compares SHA-256 to the previous snapshot.
3. If different: logs the change, extracts constants, runs diff against
   the prior snapshot, and commits the new entry to `timeline.md`.

## Files

- [`timeline.md`](timeline.md) — chronological log of all observed rolls
- [`constants_history.json`](constants_history.json) — `{date: constants}` table
- [`diffs/<date>.md`](diffs/) — one diff per roll, generated by the workflow

## Results so far

Based on the worklog up to 2026-05-20:

- iFood SDK has been on the same algorithm-core for 3+ years
- Obfuscation surface (hQ dictionary, name table) rotates ~quarterly
- Constants (TAG, FT) change with each roll
- New fields added to EV2 occasionally (1-2 per year)

## Interpretation

PX prioritizes raising the **cost of reverse engineering** (rotate obfuscation)
over changing **the algorithms** (which would require customer-side updates).
This is a rational engineering tradeoff but means a generator built once
stays mostly valid.

For defenders: if you depend on PX algorithm rotation for your defense, you
should know it's slow.
