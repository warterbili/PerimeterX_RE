# Part 1 · iFood

## 1.1 接口介绍

iFood web 流量走 `cw-marketplace.ifood.com.br`（cw = consumer web）。**4 个核心接口都要 `_px3` cookie**：

| # | 端点 | 用途 | PX 评分门槛 |
|---|---|---|---|
| 1 | `POST /v2/cardstack/search/home?lat&lng&channel=IFOOD&alias=...` | 首页 feed（按地理位置返回商家列表） | **高**（实测纯算 _px3 不够，返回 px-captcha） |
| 2 | `POST /v1/merchant-info/graphql?lat&lng&channel=IFOOD` | 商家详情 GraphQL | **中**（实测纯算 _px3 通过 ✅） |
| 3 | `GET /v1/merchants/restaurant/{id}/catalog?lat&lng` | 菜单/商品列表 | 中（推测同 #2） |
| 4 | `GET /v1/merchants/restaurant/{id}/items/{item_id}` | 商品规格/选项 | 中（推测同 #2） |

**对比**：`marketplace.ifood.com.br`（无 `cw-`，移动 API）**完全无 PX**，但要移动端 fingerprint。本研究专注 web 路径。

## 1.2 风控架构（iFood 服务侧 3 层叠加）

```
请求进来
   │
   ▼
[Layer 1] Cloudflare WAF
           IP 地理过滤 / ASN 评分 / Rate Limit
           ❌ 不通过 → HTML "Access denied" + class "no-js"
   │
   ▼ (BR 巴西住宅 IP 才能过)
   │
[Layer 2] PerimeterX
           验证 _px3 cookie 评分
           ❌ 不通过 → HTML px-captcha (含 window._pxAppId 等 JS)
   │
   ▼ (合法 _px3 + UA 一致才能过)
   │
[Layer 3] Akamai Bot Manager
           发 ak_bmsc cookie 做行为跟踪（不主动拦，用于评分）
   │
   ▼
iFood 后端业务逻辑 (37ms 处理 by x-envoy-upstream-service-time)
   │
   ▼
   200 + 真数据
```

**辨认拦在哪层**：
- HTML 含 `<title>Access to this page has been denied</title>` + `no-js` → **Cloudflare**
- HTML 含 `<meta name="description" content="px-captcha">` + `_pxAppId` → **PX**
- Response 含 `set-cookie: ak_bmsc=...` → **Akamai BMP**（通常跟 200 一起）

## 1.3 IP 要求

| IP 类型 | 结果 |
|---|---|
| 数据中心 IP | ❌ Cloudflare 直接 403 |
| 非 BR 住宅 IP | ❌ Cloudflare 拦（地理过滤） |
| **BR 巴西住宅 IP** | ✅ 全 3 层通过 |

实测代理：Bright Data residential，`country-br`，session rotation。

## 1.4 实战代码（已固化）

📄 `stample/ifood/px_cookie/business_api_demo.js`

核心逻辑：
1. monkey-patch `https.request` 加 HTTPS_PROXY 支持
2. 调 `require('./ifood_px3')` 拿 `{ cookie_value, state.vid, state.cts }`
3. 拼 `Cookie: _px3=...; _pxvid=...; _pxcts=...`
4. POST 到 GraphQL URL with `?latitude=-23.5505&longitude=-46.6333&channel=IFOOD`

## 1.6 真 PX 研究 insights

### Insight A：iFood 服务侧 3 层叠加（之前没意识到 Akamai 也在）

`set-cookie: ak_bmsc=...` 是 Akamai BMP 的 fingerprint cookie。意味着即使纯算 _px3 让 PX 评分通过，Akamai BMP 也在背后默默给请求打分。**目前我们的合法 _px3 + 合法 UA + 合法 BR IP 让 Akamai 也接受了**，但未来 Akamai 评分门槛升高的话可能要单独应对。

### Insight B：iFood 不同端点 PX 评分门槛分级

实测同一个 `_px3`，两个端点表现不同：

| 端点 | 结果 |
|---|---|
| `/v1/merchant-info/graphql` | ✅ HTTP 200 |
| `/v2/cardstack/search/home` | ❌ HTML px-captcha 403 |

**结论**：iFood 后端按端点 risk 分级 score 门槛。商家详情接受我们的纯算 _px3，但首页 feed 需要更高评分（很可能要走 Bundle 路径）。**这是一个"per-endpoint PX 评分门槛"的实证案例**。

# Part 2 · Grubhub

## 2.1 完整请求链路（账号注册/登录 9 步）

Grubhub PX **不守数据接口**（grocery.grubhub.com 用 `__Host-instacart_sid`，无 PX）。
PX **守账号体系**：注册/登录/OTP/SSO。完整注册链路：

```
[1] POST api-gtm.grubhub.com/auth (scope=anonymous)           ← PX ★
       Body: { brand: GRUBHUB, client_id, device_id, scope: anonymous }
       Response: { session_handle: { access_token: <anon UUID> }}

[2] POST api-gtm.grubhub.com/credentials                       ← PX ★★
       (要 Web Unlocker 代理，普通住宅过不了 — 这一关 PX 评分要求最高)
       Body: { email, password, first_name, last_name, ... }
       Response: 463 + verify_id (要 OTP) 或 200 (信任 IP)

[3] POST api-gtm.grubhub.com/auth/confirmation_code (直连)     ← PX ★
       Body: { email }
       Response: { csrf_token } + 发 OTP 邮件

[4] mail.tm 轮询邮件                                            ← 无 PX
       拿 OTP

[5] PUT  api-gtm.grubhub.com/auth/confirmation_code            ← PX ★
       Body: { email, code, csrf_token }
       Response: 200 + access_token + refresh_token ✅

[6] GET  grocery.grubhub.com/rest/sso/auth/awe/init             ← 无 PX
       → 302 redirect

[7] GET  api-gtm.grubhub.com/oauth2/authorize?client_id=...    ← PX ★
       → 302 redirect

[8] PUT  api-gtm.grubhub.com/oauth2/{ud_id}/access             ← PX ★
       Headers: Authorization: Bearer <access_token>
       Body: { device_id, email, password }
       Response: { status: "GRANTED", redirect_uri: "...?code=xxx" }

[9] GET  grocery.grubhub.com/rest/sso/auth/awe/callback?code   ← 无 PX
       → Set-Cookie: __Host-instacart_sid=v2.xxx ✅  ⭐ 终极产物
```

**5 个端点过 PX**：Step 1, 2, 3, 5, 7, 8（全在 api-gtm.grubhub.com）。
**1 个端点过 PX 但门槛最高**：Step 2 `/credentials`（要 Web Unlocker 代理）。

**登录 flow（已注册账号）** = Step 1 → 2'（用 /auth/login 替代 /credentials）→ 7 → 8 → 9，更短。

## 2.2 风控架构

```
请求进来 (api-gtm.grubhub.com)
   │
   ▼
[Layer 1] PerimeterX
           验证 _px2 cookie 评分
           ❌ → 403
   │
   ▼ (合法 _px2)
   │
[Layer 2] Grubhub 业务层风控
           - 同账号 + 不熟悉 device_id → 463 + 要 OTP
           - 同账号 + 熟悉 device_id → 200 + access_token
           - account locked / IP blacklist → 401/403
   │
   ▼
   业务响应
```

**关键认知 — HTTP 463 vs 403 区分**：
- HTTP **403** = PX 拦截（_px2 评分不够，根本没到业务层）
- HTTP **463** = PX 通过 + Grubhub 业务层风控要 OTP
- HTTP **200** = PX + 业务层都通过

463 算 **PX-pass**（桌面 (3) 项目 5/5 真账号 bootstrap 同样判定）。

## 2.3 IP 要求

| IP 类型 | 结果 |
|---|---|
| 本机 IP（国内非 US） | ✅ Stage 1+2+3 全通过（实测！） |
| US 住宅 IP | ✅（推荐 — Grubhub 是美国站，PX 评分更高） |
| 数据中心 IP | ⚠️ 评分降低，可能 403 |

**Grubhub 比 iFood 宽松很多** —— 本机 IP 就能跑通 2 个 PX 守门端点。

## 2.4 实战代码（已固化）

📄 `stample/grub/px_cookie/business_api_demo.js`

核心逻辑：
1. monkey-patch `https.request` 加 HTTPS_PROXY 支持
2. 调 `require('./grubhub_px2')` 拿 `_px2` cookie
3. **Stage 1**: POST `/auth (anonymous)` → 拿 anon_token
4. **Stage 2**（可选，由 env 控制）: POST `/auth/login` with Bearer anon_token + email/password → 200 or 463

凭据全走 env var（`GRUBHUB_EMAIL` / `GRUBHUB_PASSWORD`），不嵌代码。

**含义**：PX 通过 ✅，Grubhub 业务层因 device_id 不熟悉要求 OTP。要拿 200 完整登录需要走 Step 4-5（mail.tm OTP fetch + PUT /auth/confirmation_code），本 demo 不涵盖。

## 2.6 真 PX 研究 insights

### Insight C：HTTP 463 = PX-pass

PX 视角看，463 是**业务层 reject**而非 PX reject。判定标准：
- 是否含 `verify_methods` field → 业务层风控发起的 OTP → PX 已通过
- response 是 JSON 不是 HTML → 没被 PX captcha 中间页拦截

### Insight D：多端点 PX 一致评分

`_px2` 一旦生成，在 Grubhub 多个 PX 端点（/auth + /auth/login）评分一致。**跟 iFood 不同**（iFood 不同端点门槛不同）。这是 Grubhub 比 iFood 简单的关键 — 拿一个 _px2 整个账号链路都能用。
