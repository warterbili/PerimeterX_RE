# Part 1 · iFood

## 1.1 API Overview

iFood web traffic goes through `cw-marketplace.ifood.com.br` (cw = consumer web). **All 4 core APIs require the `_px3` cookie**:

| # | Endpoint | Purpose | PX score threshold |
|---|---|---|---|
| 1 | `POST /v2/cardstack/search/home?lat&lng&channel=IFOOD&alias=...` | Home feed (geographic merchant list) | **High** (pure-algo _px3 insufficient in practice, returns px-captcha) |
| 2 | `POST /v1/merchant-info/graphql?lat&lng&channel=IFOOD` | Merchant details GraphQL | **Medium** (pure-algo _px3 passes ✅) |
| 3 | `GET /v1/merchants/restaurant/{id}/catalog?lat&lng` | Menu / product list | Medium (presumed same as #2) |
| 4 | `GET /v1/merchants/restaurant/{id}/items/{item_id}` | Product specifications / options | Medium (presumed same as #2) |

**Comparison**: `marketplace.ifood.com.br` (no `cw-`, mobile API) **has no PX at all**, but requires mobile fingerprints. This study focuses on the web path.

## 1.2 Risk Architecture (3 Server-side Layers on iFood)

```
incoming request
   │
   ▼
[Layer 1] Cloudflare WAF
           IP geo filter / ASN scoring / Rate Limit
           ❌ blocked → HTML "Access denied" + class "no-js"
   │
   ▼ (only BR Brazilian residential IP passes)
   │
[Layer 2] PerimeterX
           Validates _px3 cookie score
           ❌ blocked → HTML px-captcha (contains window._pxAppId etc. JS)
   │
   ▼ (legitimate _px3 + consistent UA passes)
   │
[Layer 3] Akamai Bot Manager
           Issues ak_bmsc cookie for behavioral tracking (doesn't actively block; used for scoring)
   │
   ▼
iFood backend business logic (37ms processing per x-envoy-upstream-service-time)
   │
   ▼
   200 + real data
```

**Identify which layer rejected**:
- HTML contains `<title>Access to this page has been denied</title>` + `no-js` → **Cloudflare**
- HTML contains `<meta name="description" content="px-captcha">` + `_pxAppId` → **PX**
- Response contains `set-cookie: ak_bmsc=...` → **Akamai BMP** (usually accompanies 200)

## 1.6 PX Research Insights

### Insight A: iFood Stacks 3 Server-side Layers (Previously Unaware Akamai Was Also Present)

`set-cookie: ak_bmsc=...` is Akamai BMP's fingerprint cookie. It means even when pure-algo _px3 passes PX scoring, Akamai BMP is quietly scoring the request in the background. **Currently our legitimate _px3 + legitimate UA + legitimate BR IP makes Akamai accept us too**, but if Akamai's scoring threshold rises in the future, we may need to handle it separately.

### Insight B: iFood Stratifies PX Score Thresholds Per Endpoint

Measured with the same `_px3`, two endpoints behave differently:

| Endpoint | Result |
|---|---|
| `/v1/merchant-info/graphql` | ✅ HTTP 200 |
| `/v2/cardstack/search/home` | ❌ HTML px-captcha 403 |

**Conclusion**: iFood's backend tiers score thresholds by endpoint risk. Merchant details accept our pure-algo _px3, but the home feed needs a higher score (very likely requires the Bundle path). **This is an empirical case of "per-endpoint PX score threshold"**.

# Part 2 · Grubhub

## 2.1 Complete Request Chain (Account Registration / Login — 9 Steps)

Grubhub's PX **does not gate data interfaces** (grocery.grubhub.com uses `__Host-instacart_sid`; no PX).
PX **gates the account system**: registration / login / OTP / SSO. Full registration chain:

```
[1] POST api-gtm.grubhub.com/auth (scope=anonymous)           ← PX ★
       Body: { brand: GRUBHUB, client_id, device_id, scope: anonymous }
       Response: { session_handle: { access_token: <anon UUID> }}

[2] POST api-gtm.grubhub.com/credentials                       ← PX ★★
       (requires Web Unlocker proxy; plain residential won't pass — highest PX score threshold here)
       Body: { email, password, first_name, last_name, ... }
       Response: 463 + verify_id (needs OTP) or 200 (trusted IP)

[3] POST api-gtm.grubhub.com/auth/confirmation_code (direct)   ← PX ★
       Body: { email }
       Response: { csrf_token } + sends OTP email

[4] mail.tm polls mailbox                                       ← no PX
       Obtain OTP

[5] PUT  api-gtm.grubhub.com/auth/confirmation_code            ← PX ★
       Body: { email, code, csrf_token }
       Response: 200 + access_token + refresh_token ✅

[6] GET  grocery.grubhub.com/rest/sso/auth/awe/init             ← no PX
       → 302 redirect

[7] GET  api-gtm.grubhub.com/oauth2/authorize?client_id=...    ← PX ★
       → 302 redirect

[8] PUT  api-gtm.grubhub.com/oauth2/{ud_id}/access             ← PX ★
       Headers: Authorization: Bearer <access_token>
       Body: { device_id, email, password }
       Response: { status: "GRANTED", redirect_uri: "...?code=xxx" }

[9] GET  grocery.grubhub.com/rest/sso/auth/awe/callback?code   ← no PX
       → Set-Cookie: __Host-instacart_sid=v2.xxx ✅  ⭐ ultimate artifact
```

**5 endpoints go through PX**: Steps 1, 2, 3, 5, 7, 8 (all on api-gtm.grubhub.com).
**1 endpoint goes through PX with the highest threshold**: Step 2 `/credentials` (requires Web Unlocker proxy).

**Login flow (already-registered account)** = Step 1 → 2' (use /auth/login instead of /credentials) → 7 → 8 → 9; shorter.

## 2.2 Risk Architecture

```
incoming request (api-gtm.grubhub.com)
   │
   ▼
[Layer 1] PerimeterX
           Validates _px2 cookie score
           ❌ → 403
   │
   ▼ (legitimate _px2)
   │
[Layer 2] Grubhub business-layer risk control
           - Same account + unfamiliar device_id → 463 + OTP required
           - Same account + familiar device_id → 200 + access_token
           - account locked / IP blacklisted → 401/403
   │
   ▼
   Business response
```

**Key insight — HTTP 463 vs 403 distinction**:
- HTTP **403** = PX rejection (_px2 score insufficient; didn't reach business layer)
- HTTP **463** = PX pass + Grubhub business-layer requires OTP
- HTTP **200** = Both PX and business layer pass

463 counts as **PX-pass** (Desktop (3) project's 5/5 real-account bootstraps reach the same verdict).

## 2.3 IP Requirements

| IP type | Result |
|---|---|
| Local IP (non-US country) | ✅ Stages 1+2+3 all pass (measured!) |
| US residential IP | ✅ (recommended — Grubhub is US-based; PX score higher) |
| Datacenter IP | ⚠️ Score lowered; may 403 |

**Grubhub is much more lenient than iFood** — local IP suffices to pass 2 PX-gated endpoints.

## 2.4 Production Code (Committed)

📄 `stample/grub/px_cookie/business_api_demo.js`

Core logic:
1. monkey-patch `https.request` to support HTTPS_PROXY
2. call `require('./grubhub_px2')` to get `_px2` cookie

## 2.6 PX Research Insights

### Insight C: HTTP 463 = PX-pass

From PX's perspective, 463 is a **business-layer reject**, not a PX reject. Discriminators:
- Contains `verify_methods` field → OTP initiated by business-layer risk control → PX has passed
- Response is JSON not HTML → not blocked by PX captcha intermediary page

### Insight D: Multi-endpoint PX Scores Consistently

Once `_px2` is generated, it scores consistently across Grubhub's multiple PX-gated endpoints (/auth + /auth/login). **Different from iFood** (different endpoints have different thresholds). This is the key reason Grubhub is simpler than iFood — one _px2 covers the entire account chain.
