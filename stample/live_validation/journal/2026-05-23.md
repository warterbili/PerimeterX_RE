# Part 2 · Grubhub

## 2.1 关键发现 —— 5 件叠死

之前公开 demo 卡在 `/auth/login` HTTP 403（PX-Bundle 挑战），定位为以下 5 处同时错：

| # | 错误 | 修复 |
|---|---|---|
| 1 | 路径选错：直接打 `/auth/login`（PX 高门槛） | **优先 `/auth/refresh`** —— 无 PX 挑战，refresh 失败才 fallback /auth/login |
| 2 | UA `Chrome 148` 跟 `impersonate='chrome120'` 不匹配 | UA 改 **Chrome 120**，跟 TLS 指纹对齐 |
| 3 | SSO step 3 加 `perimeter-x` / `sec-ch-ua-*` header | **裸 `_GH_HEADERS` + Bearer**，加 PX header 反而 HMAC 不匹配 → 403 |
| 4 | `device_id` 用 integer `-981990071` | **用 string `"-981990071"`**（业务层格式校验严；该数跟具体账号绑定，错一位 → 463） |
| 5 | 账号源 `accounts.jsonl`（60 个 stale）| 用 `_bootstrap_5.jsonl`（5 个本地维护账号；其中 2 个 device 已被 Grubhub 业务层信任，跳过 OTP）|

**单独修任一处都还是死**，必须 5 处全套用上才能通。

## 2.2 完整 9 步链路（今天实战）

```
[1] 生成 _px2                      px_pure_cookie_v2
[2] POST /auth (anonymous)        HTTP 200  ← PX 守门 #1 ✓
[3] POST /auth/refresh OR /login  HTTP 200  ← refresh 优先；fallback login 时 device 信任直接 200
[4-6] (OTP 路径，本次未触发)
[7] GET  grocery.../awe/init      HTTP 302  ← SSO 1
[8] GET  api-gtm.../authorize     HTTP 303  ← SSO 2
[9] PUT  api-gtm.../oauth2/<ud_id>/access  HTTP 200  ← SSO 3 (此步今天 PX 也守，裸 headers 过)
[10] GET grocery.../awe/callback  ← Set-Cookie: __Host-instacart_sid ⭐
```

## 2.5 真 PX 研究 insights

### Insight A：`/auth/refresh` 是 PX 旁路

下游生产项目把入口顺序改成「refresh 优先，login fallback」之后，refresh_token 活的账号可以**完全绕过 PX 高门槛端点**，直接 `/auth/refresh` 200 → SSO → done。

### Insight B：SSO step 3 PX 化 + 裸 header 反而过

之前 journal §2 标注 SSO step 3 (`PUT /oauth2/<ud_id>/access`) 不需要 PX。今天实测它**已经被 PX 守住**，但奇怪的是带 `perimeter-x` header 反而 403（HMAC 校验不通过），只用 `_GH_HEADERS + Bearer` 裸跑反而 200。结论：PX 在这个端点只校验 cookie + IP 信誉，不再读 header。

### Insight C：UA / impersonate 必须严格匹配

`curl_cffi.Session(impersonate='chrome120')` 设定的是 TLS / HTTP/2 帧序列指纹，但 **User-Agent header 是独立维度**。两者必须对齐到同一个 Chrome 版本（120 / 145 / …），否则 PX 直接打 bot 标签 → 全链路任何端点都可能 403。

### Insight D：mail.tm 域 ≠ 全部账号

3/5 失败账号触发 463 但 mail.tm 拿不到 JWT —— 因为 mail.tm 上的密码跟 Grubhub 上的密码不同步（可能 mail.tm 那侧账号被删 / 密码重置）。批处理的容错策略：**拿不到 JWT 就直接 fail 当前账号，不阻塞其它账号**。
